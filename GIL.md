**GIL (Global Interpreter Lock)** — механизм в [[Реализации интерпретатора Python|CPython]], который гарантирует, что в один момент времени байт-код Python исполняет только один поток, даже если процесс использует несколько потоков ОС. Это делает объектную модель CPython (включая встроенные типы, например `dict`) неявно безопасной при конкурентном доступе из нескольких потоков — без GIL пришлось бы расставлять блокировки на каждый объект отдельно.

## Зачем он нужен

CPython управляет памятью объектов через счётчик ссылок (reference counting): у каждого объекта есть поле `ob_refcnt`, которое увеличивается и уменьшается при создании/удалении ссылок. Без блокировки два потока могли бы одновременно изменять этот счётчик у одного объекта — гонка данных привела бы к утечкам памяти или преждевременному удалению объекта, на который ещё есть ссылки.

GIL — самое простое решение этой проблемы: вместо тонких блокировок на каждый объект (что замедлило бы однопоточный код и усложнило бы C-расширения) используется одна глобальная блокировка на весь интерпретатор.

## Как это работает

- Поток, чтобы исполнять байт-код Python, должен сначала захватить GIL.
- Интерпретатор периодически принудительно освобождает GIL, давая шанс исполниться другим потокам — начиная с Python 3.2 это происходит не по числу выполненных байт-код инструкций, а по интервалу времени (`sys.setswitchinterval()`, по умолчанию 5 мс).
- При операциях I/O (чтение файла, сетевой запрос, `time.sleep`) поток явно отпускает GIL на время ожидания — поэтому I/O-bound многопоточность в Python работает эффективно.
- C-расширения (например, NumPy при тяжёлых вычислениях) могут явно освобождать GIL на время выполнения C-кода, не трогающего объекты Python.

## Последствия для многопоточности

| Тип задачи | Эффект от `threading` |
|---|---|
| I/O-bound (сеть, диск, ожидание) | Потоки эффективно параллелятся — GIL освобождается на время ожидания I/O |
| CPU-bound (вычисления на чистом Python) | Параллелизма нет — потоки исполняются по очереди, реального ускорения от многопоточности не будет |

```python
# ❌ CPU-bound задача в потоках не ускоряется из-за GIL
import threading

def count(n):
    while n > 0:
        n -= 1

threads = [threading.Thread(target=count, args=(10**7,)) for _ in range(4)]
# выполнение этих потоков не быстрее, чем последовательный вызов count() 4 раза
```

## Как обойти ограничение GIL

- **`multiprocessing`** — каждый процесс получает свой интерпретатор и свой GIL, реальный параллелизм для CPU-bound задач за счёт отдельных процессов (с накладными расходами на межпроцессное взаимодействие).
- **C-расширения, освобождающие GIL** — NumPy, многие операции с массивами выполняются в C без удержания GIL.
- **Альтернативные реализации без GIL** — [[Реализации интерпретатора Python|Jython и IronPython]] не имеют GIL (управляют потоками через JVM/CLR соответственно).
- **PEP 703** — экспериментальная возможность собрать CPython без GIL (free-threaded build), доступна начиная с Python 3.13 как опциональный режим, пока не используется по умолчанию.

## GIL — особенность реализации, а не языка

GIL присутствует в CPython, но это не требование спецификации языка Python: [[Реализации интерпретатора Python|Jython и IronPython]] обходятся без него, управляя потоками через JVM и CLR соответственно.
