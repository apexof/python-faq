#### Где хранятся объекты и переменные?
Сами объекты — в куче. 
Локальные переменные — на стеке (во [[Стек вызовов|фреймах]] функций) 
Глобальные переменные — в словаре модуля `globals()`, который тоже лежит в куче.

#### Как устроена память в питоне?

**1. Все - объекты**
Переменные хранят ссылки на объекты, а не сами значения.
Объект содержит: значение, тип, счётчик ссылок.

**2. Reference Counting**
Каждый объект имеет счётчик ссылок.
Когда счётчик = 0, память освобождается.

```python
import sys
sys.getrefcount(a)  # показывает количество ссылок
```
#### Что такое garbage collector?
Garbage collector (GC) — автоматическая система удаления объектов, на которые больше никто не ссылается. Находит и удаляет циклические ссылки (A → B → A).  Трекает только контейнеры (`list`, `dict`, `set`, экземпляры классов), способные образовать цикл; атомарные объекты вроде чисел и строк игнорирует.

#### Как работает garbage collector?
- Три поколения и их пороги: gen0 - 700, gen1 - 10,  gen2 - 10
- Новый объект попадает в **поколение 0**.
- GC ведёт счётчик: `count = (аллокации − деаллокации)` трекаемых объектов.
- Когда `count[0] > 700` → **сборка поколения 0**.
- Каждые 10 сборок gen0 → **сборка gen1**.
- Каждые 10 сборок gen1 → **сборка gen2** (полная, самая дорогая).
- Объект, переживший сборку своего поколения, переходит в следующее 
- Старые объекты проверяются реже, новые чаще

#### Как управлять garbage collector вручную?
Модуль `gc`:
- `gc.collect()` — запустить сборку принудительно, возвращает число собранных объектов.
- `gc.disable()` / `gc.enable()` — выключить/включить автоматический запуск сборщика (reference counting продолжает работать всегда, отключается только поиск циклов).
- `gc.get_threshold()` — пороги поколений: `(700, 10, 10)`.
- `gc.get_count()` — текущие счётчики по поколениям.

```python
import gc
gc.get_threshold()  # (700, 10, 10)
gc.disable()
gc.collect()         # можно вызвать и вручную, даже при отключённом авто-запуске
```

#### Что такое gc.garbage и почему объект может туда попасть?
`gc.garbage` — список объектов, которые сборщик нашёл в цикле, но не смог освободить безопасно. До PEP 442 (Python 3.4) это касалось объектов с `__del__`, замешанных в цикл: порядок вызова деструкторов в цикле не определён, поэтому сборщик боялся вызывать `__del__` и складывал такие объекты в `gc.garbage`, не освобождая. После PEP 442 GC умеет финализировать и такие циклы, поэтому на современных версиях Python `gc.garbage` в этом сценарии обычно пуст — там остаются только настоящие патологические случаи (например, объект оживает в своём `__del__`).

#### Как избежать циклических ссылок без участия GC?
`weakref` — слабая ссылка на объект, которая не увеличивает счётчик ссылок и не мешает `reference counting` освободить объект.

```python
import weakref
f = object()
r = weakref.ref(f)
r()      # объект, пока жив
del f
r()      # None — объект уже удалён
```
#### Интернирование
Интернирование - переиспользование объектов в памяти для разных переменных.
Объекты в памяти создаются заранее, при создании переменной новый объект не создается, ссылка будет указывать на уже созданный объект.

Что интернируется:
- Числа: -5 … 256
- Строки: 
	- Строки-литералы, похожие на идентификаторы (буквы, цифры, `_`)
	- Имена переменных, функций, классов, атрибутов
	- Пустая строка и односимвольные
	- Константы, свёрнутые на компиляции
	- НЕ интернируются: с пробелами/спецсимволами и созданные в рантайме
- bool — True, False: всегда один объект
- None: единственный экземпляр в программе, поэтому проверяют `x is None`.
- пустой tuple

#### Free lists (Свободные списки)
Free lists - пулы переиспользуемой памяти для часто создаваемых объектов (tuple, list, dict, float, [[Стек вызовов|frames]]…)
Когда объект удаляется из памяти, CPython не возвращает его память ОС, а кладёт «пустой слот» в список — free list. При следующем создании объекта того же типа берётся готовый слот из пула, и в него записывается новое содержимое.

#### Constant folding
Constant folding — оптимизация на этапе компиляции: вычисление выражений из констант заранее, чтобы не считать их каждый раз в рантайме. Результат сразу кладётся в байткод.

```python
x = 2 + 3        # компилятор сразу пишет x = 5
s = "ab" * 3     # → "ababab"
```

В байткоде уже лежит готовая константа, а не инструкции сложения.

**Что сворачивается:**
- арифметика над литералами: `2 + 3`, `10 ** 2`
- операции над строками/байтами: `"a" + "b"`, `"x" * 5`
- кортежи из констант: `(1, 2, 3)`
- булева/сравнения констант в некоторых случаях

**Что НЕ сворачивается:**
- переменные: `x + 3` — значение `x` неизвестно при компиляции
- слишком большие результаты: `2 ** 100000` — иначе раздует байткод и замедлит старт (есть лимит)
- вызовы функций: `len("abc")` — функция могла быть переопределена

#### Алокатор
**Аллокатор (allocator)** — это компонент, который управляет выделением и освобождением памяти: по запросу выдаёт блок нужного размера, а при освобождении забирает обратно для повторного использования.

В Python объекты создаются постоянно и в основном мелкие (int, маленькие строки, узлы структур). Звать системный `malloc`/`free` напрямую на каждый объект — дорого. Поэтому CPython ставит свой аллокатор поверх системного.
