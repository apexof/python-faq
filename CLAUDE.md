# Python Конспекты — CLAUDE.md

Это хранилище материалов по Python-разработке в формате вопрос/ответ, написанных в Obsidian, для подготовки к техническому собеседованию.

## Цель

- Приоритет — темы и формулировки, которые реально звучат на собесе, а не редкая экзотика языка.
- Если у понятия есть устоявшийся термин на собесе (например, "duck typing", "late binding") — использовать его, а не придумывать русский аналог.
- Примеры кода — такие, какие могут попросить написать на собесе устно или на бумаге: рабочие, минимальные, без лишней инфраструктуры.

## Источники информации

Для проверки фактов использовать только достоверные источники, в порядке приоритета:
1. **Официальная документация Python** — `docs.python.org/3/`. В первую очередь:
   - `library/stdtypes.html` — встроенные типы.
   - `library/types.html` — служебные типы (`FunctionType`, `GeneratorType`, `ModuleType`...).
   - `reference/datamodel.html` — объектная модель, специальные методы.
   - `library/exceptions.html` — иерархия исключений.
2. **PEP** (`peps.python.org`) — когда нужно объяснить, почему синтаксис/поведение появилось именно так (например, PEP 8, PEP 484 для тайпинга, PEP 525 для async generators).
3. **Реальное поведение интерпретера** — если документация неоднозначна, проверить через `python3 -c "..."` и зафиксировать результат как факт, а не как предположение.

**Что не считается достоверным источником:** ответы по памяти без проверки, блоги и статьи без перепроверки через документацию. Если факт не проверен — явно отмечать это (например: "точное число не зафиксировано в документации, это агрегированная оценка"), а не выдавать предположение за цитату.

## Формат файла

Каждый файл — набор вопросов по одной теме. Один вопрос — один блок:

```markdown
#### Текст вопроса?
Ответ или список пунктов.
```

- Заголовок вопроса — всегда `####`.
- Если ответ — несколько фактов одного уровня, оформлять списком `-`, без вложенности.
- Если ответ — рассуждение или вывод, писать обычным абзацем.
- Между вопросами — одна пустая строка.
- Код — в блоке ` ```python` с указанием языка, с пустой строкой до и после блока.
- Порядок вопросов в файле — по смыслу (от определения и базовых свойств к механике, деталям и частным случаям), а не по хронологии добавления.
- Эталон форматирования — `Вопросы/Вопросы про список list.md`. При сомнениях ориентироваться на него.

## Язык

- Весь текст — на русском, термины и код — на английском.
- Смешивать языки в одном предложении — нормально: "**Comprehension** — это синтаксис Python для...".
- Устоявшиеся англоязычные термины (из документации, концепции типа "duck typing") не переводить — так легче сопоставить с источником и с собесом.

## Чего не делать

- Не писать мета-фразы про собес внутри ответа (например "это часто спрашивают на собесе").
- Не добавлять раздел "Смотрите также" / "Связанные разделы".
- Не дублировать информацию из других файлов — ставить ссылку `[[ИмяФайла]]`.
- Не давать краткий список, а потом расширять то же самое ниже — выбрать один формат ответа и придерживаться его.
- Не выдавать непроверенный факт за цитату из документации.

## Эталон стиля

Компактно, по делу, без воды, без повторов. Каждый новый файл — такой же.
