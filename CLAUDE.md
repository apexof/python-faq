# Python Конспекты — CLAUDE.md

Это хранилище конспектов по Python-разработке, написанных в Obsidian.

## Цель конспекта

Конспекты пишутся **для подготовки к техническому собеседованию по Python**. Из этого следует:
- Приоритет — темы и формулировки, которые реально звучат на собесе, а не редкая экзотика языка.
- Если у понятия есть устоявшийся термин на собесе (например, "duck typing", "late binding") — использовать его, а не придумывать русский аналог.
- Примеры кода должны быть такими, какие могут попросить написать на собесе устно или на бумаге — рабочими, но без лишней инфраструктуры.

## Источники информации

Для проверки фактов использовать только достоверные источники, в порядке приоритета:
1. **Официальная документация Python** — `docs.python.org/3/`. В первую очередь:
   - `library/stdtypes.html` — встроенные типы.
   - `library/types.html` — служебные типы (`FunctionType`, `GeneratorType`, `ModuleType`...).
   - `reference/datamodel.html` — объектная модель, специальные методы.
   - `library/exceptions.html` — иерархия исключений.
2. **PEP** (`peps.python.org`) — когда нужно объяснить, почему синтаксис/поведение появилось именно так (например, PEP 8, PEP 484 для тайпинга, PEP 525 для async generators).
3. **Реальное поведение интерпретера** — если документация неоднозначна, проверить через `python3 -c "..."` и зафиксировать результат как факт, а не как предположение.

**Что не считается достоверным источником:** ответы по памяти без проверки, блоги и статьи без перепроверки через документацию. Если факт не проверен — явно отмечать это (например: "точное число не зафиксировано в документации, это агрегированная оценка"), а не выдавать предположение за цитату.

## Структура папок

Темы разбиты по технологиям и областям знаний:

```
Python/
├── Типы данных/          # int, str, list, dict и другие типы значений
│   └── Служебные и редкие типы.md  # slice, code, frame и др. — одним файлом
├── Функции/             # Функции, аргументы, замыкания, декораторы, генераторы
├── ООП/                 # Классы, наследование, магические методы, объектная модель
├── Структуры данных/     # Списки, словари, множества, кортежи
│   └── Comprehensions/   # Подтемы — отдельная папка
├── Асинхронность/        # asyncio, корутины, event loop
├── Исключения/           # Иерархия ошибок, try/except, свои исключения
├── Стандартная библиотека/ # Модули stdlib: os, itertools, collections...
├── Тестирование/          # pytest, unittest, моки
├── Окружение/             # venv, pip, poetry, управление зависимостями
└── DevTeam/               # Дополнительные конспекты (черновики / старые)
```

**Принцип:** одна тема — один файл. Если подтем много, создаётся папка с файлом-оглавлением (того же названия, что и папка) и отдельными файлами для каждой подтемы.

### Разделение смыслов слова "тип"
В Python слово "тип" означает разные вещи на разных уровнях — это не должно сваливаться в одну папку:
- **Тип данных** (значение, которое хранит переменная: `int`, `list`, `dict`...) → папка `Типы данных/`.
- **Тип объекта объектной модели** (всё в Python — объект, поэтому у функции/класса/модуля тоже есть тип: `function`, `type`, `module`...) → тематическая папка, где это изучается (`ООП/`, `Функции/`, `Асинхронность/`).
- **Тип ошибки** (`BaseException` и иерархия) → `Исключения/`.
- Низкоуровневая служебка языка (`slice`, `code`, `frame`, `traceback`...), которая не является учебным материалом первого порядка, — не отдельные файлы, а один файл-приложение `Служебные и редкие типы.md`.

## Стиль написания

### Язык
- Весь текст **на русском**, термины и код — на английском.
- Смешивать языки в одном предложении — нормально: "**Comprehension** — это синтаксис Python для...".
- Устоявшиеся англоязычные названия (термины из документации, заголовки разделов вроде "Built-in Types", "Data model", названия концепций типа "duck typing") можно оставлять как есть, не переводить — так их легче сопоставить с источником и с собесом, где их называют по-английски.

### Структура файла
- Первая строка — **определение** жирным шрифтом или обычным текстом: что это такое, одной фразой.
- Далее — разделы `##` с конкретными аспектами темы.
- Разделы короткие и конкретные, без лишней воды.

### Форматирование
- Активно используются таблицы для сравнений и перечней.
- Примеры кода — с комментариями `# ✅ правильно` / `# ❌ неправильно` прямо в коде.
- Эмодзи ✅ ❌ ⚠️ для пометок — это норма и приветствуется.
- Заголовки с нумерацией — когда важен порядок. Уровень (`##` или `####`) выбирается по визуальной компактности в Obsidian: `####` даёт меньше отступов и подходит для коротких конспектов.

### Ссылки (Obsidian wikilinks)
- Ссылки на связанные файлы через `[[ИмяФайла]]`.
- Если текст ссылки отличается от имени файла: `[[текст|ИмяФайла]]`.
- Ссылки ставятся inline в тексте при первом упоминании термина.
- **Не добавлять** разделы "Смотрите также" или "Связанные разделы" в конце файла.

### Примеры кода
- Всегда указывать язык в блоке кода: ` ```python`, ` ```shell`.
- Примеры — рабочие и минимальные, без лишнего обвеса.
- Показывать и правильный, и неправильный варианты когда это уместно.

## Глубина проработки темы

- Цель — разобраться **глубоко**, а не зафиксировать поверхностный факт. Конспект "на отвали" (1-2 строки без объяснения механики) — недостаточен.
- Если у темы есть распространённые/редкие подтипы или варианты (например: типы данных, виды исключений, виды аргументов функции) — раскрывать **все**, а не только базовые/частые. "Базовые" не значит "единственные".
- Для каждого пункта — не просто название, а что это, как ведёт себя, чем отличается от похожих, и минимальный пример кода, если это применимо.
- Лучше один полный файл по теме, чем несколько поверхностных по подтемам — если того требует принцип "одна тема — один файл".

## Чего не делать

- Не добавлять раздел "Смотрите также" / "Связанные разделы".
- Не писать длинные введения — сразу к сути.
- Не дублировать информацию из других файлов — ставь ссылку.
- Не создавать файлы вне тематических папок (исключение: файлы на корневом уровне типа `Регулярные выражения.md` — для тем без подпапки).
- **Не давать краткую таблицу/список, а потом расширять то же самое ниже** — это дублирование. Выбери один формат и придерживайся его.

## Эталон стиля

Компактно, по делу, без воды, без повторов. Каждый новый файл должен быть таким же.
