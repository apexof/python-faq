**Мультипроцессинг (multiprocessing)** в Python — выполнение нескольких отдельных процессов ОС с помощью модуля `multiprocessing`. Каждый процесс получает собственный интерпретатор Python и собственный [[GIL]], поэтому процессы исполняются по-настоящему параллельно на разных ядрах CPU.

## Зачем нужен, в отличие от `threading`

[[Многопоточность|Потоки]] в [[Реализации интерпретатора Python|CPython]] ограничены [[GIL]] — параллельно исполнять байт-код Python не могут. Процессы этого ограничения не имеют, так как у каждого свой интерпретатор и своя память.

| | `threading` | `multiprocessing` |
|---|---|---|
| Параллелизм для CPU-bound задач | Нет (ограничен GIL) | Да (каждый процесс — свой GIL) |
| Память | Общая для всех потоков | Изолированная у каждого процесса |
| Накладные расходы на создание | Низкие | Выше (создание процесса дороже потока) |
| Обмен данными | Прямой доступ к общим объектам | Через сериализацию (`pickle`) — `Queue`, `Pipe`, общая память |
| Подходит для | I/O-bound задач | CPU-bound задач |

## Основные объекты модуля `multiprocessing`

| Объект | Назначение |
|---|---|
| `Process` | Класс для создания и запуска процесса |
| `Pool` | Пул процессов для параллельного выполнения функции на наборе входных данных |
| `Queue` | Очередь для обмена данными между процессами (потокобезопасная и процессобезопасная) |
| `Pipe` | Двунаправленный канал связи между двумя процессами |
| `Value`, `Array` | Объекты в разделяемой памяти для простых данных |
| `Lock`, `Event`, `Semaphore` | Аналоги примитивов синхронизации из `threading`, но для процессов |

## Создание процесса

```python
import multiprocessing

def worker(n):
    print(f"процесс {n} работает")

if __name__ == "__main__":   # обязательно на Windows и при использовании spawn
    processes = []
    for i in range(3):
        p = multiprocessing.Process(target=worker, args=(i,))
        processes.append(p)
        p.start()

    for p in processes:
        p.join()
```

`if __name__ == "__main__":` обязателен, потому что при способе старта `spawn` (используется по умолчанию на Windows и macOS) дочерний процесс заново импортирует основной модуль — без этой проверки запуск ушёл бы в бесконечную рекурсию создания процессов.

## Pool — параллельное выполнение функции

```python
from multiprocessing import Pool

def square(x):
    return x * x

if __name__ == "__main__":
    with Pool(processes=4) as pool:
        result = pool.map(square, [1, 2, 3, 4, 5])
    print(result)  # [1, 4, 9, 16, 25]
```

`Pool.map` распределяет элементы итерируемого объекта между процессами пула и возвращает результаты в исходном порядке.

## Обмен данными между процессами

Процессы не делят память, поэтому данные между ними передаются через сериализацию (`pickle`):

```python
from multiprocessing import Process, Queue

def worker(q):
    q.put("результат из процесса")

if __name__ == "__main__":
    q = Queue()
    p = Process(target=worker, args=(q,))
    p.start()
    p.join()
    print(q.get())  # "результат из процесса"
```

- Передаваемые объекты должны поддерживать `pickle` — функции, открытые файлы, сокеты и т.п. сериализовать напрямую нельзя.
- `Value` и `Array` позволяют избежать сериализации для простых типов, используя разделяемую память напрямую.

## Способы запуска процесса (start methods)

| Способ | Поведение |
|---|---|
| `fork` | Дочерний процесс — полная копия родительского (включая память); начиная с Python 3.14 не используется по умолчанию ни на одной платформе — нужно указывать явно через `set_start_method()` |
| `spawn` | Запускается новый чистый интерпретатор, импортирует модуль заново; по умолчанию на Windows всегда, на macOS — начиная с Python 3.8 |
| `forkserver` | Отдельный сервер-процесс форкает новые процессы по запросу; по умолчанию на Linux начиная с Python 3.14 (раньше там был `fork`) |

## Когда использовать `multiprocessing`

- CPU-bound вычисления на чистом Python, где [[GIL]] мешает [[Многопоточность|потокам]] дать реальное ускорение.
- Задачи, которые можно разбить на независимые части без частого обмена данными (накладные расходы на сериализацию и создание процессов выше, чем у потоков).
